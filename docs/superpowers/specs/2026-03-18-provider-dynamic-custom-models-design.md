# Provider 自定义模型动态注册设计文档

**日期**：2026-03-18
**状态**：待实施
**模块**：`app.go`

---

## 背景与问题

当前 `editProvider()` 的模型配置流程分成两条保存路径：

1. 预设模型通过 `MultiSelect` 选择，写入 `selectedModels`
2. 自定义模型通过底部输入框填写，提交后由 `mergeProviderModels()` 与预设选择结果合并

这会带来两个体验问题：

1. **自定义模型不可见**：用户输入自定义模型后，无法在上方模型列表中立即看到它，也无法在同一轮表单中继续勾选/取消
2. **列表样式不符合预期**：当前 `huh` 默认主题中，未选中项前缀为小圆点 `•`，用户希望改成更稳定直观的 ASCII 复选框样式

用户已经确认以下交互要求：

- 自定义模型在输入框失焦时（`enter/tab` 离开字段）应**立即加入上方模型列表**
- 新加入的自定义模型应**默认自动选中**
- 后续即使把输入框内容删掉，列表中的该模型也应**继续保留**
- 多选列表样式改为：未选中 `[ ]`，选中 `[✓]`
- 自定义输入框在完成注册后应**自动清空**，作为“新增入口”而非长期状态展示区

---

## 目标

在不改变 `editProvider()` 整体表单顺序的前提下，实现如下体验：

1. 用户在“自定义模型名称”输入框中输入一个或多个模型名
2. 离开该字段后，新模型立刻出现在上方 `MultiSelect` 中
3. 新模型默认处于选中状态
4. 模型是否最终保存，仅由 `MultiSelect` 当前勾选结果决定
5. 列表视觉从圆点切换为 ASCII 复选框

---

## 方案比较

### 方案 A（推荐）：动态 `MultiSelect` + 自定义模型注册池

将 `MultiSelect` 改为动态 options，统一显示：

- 预设模型
- 已保存过的自定义模型
- 本次会话中新注册的自定义模型

底部输入框不再直接参与最终保存，而是只负责把新模型注册到列表中，并自动加入当前勾选结果。

**优点**：

- 完全符合用户确认的交互
- 状态职责清晰：输入框负责新增，列表负责选择
- 支持“加入后保留”这一要求，不会因为输入框内容变化而丢失列表项

**缺点**：

- 需要引入表单内的动态状态与 `OptionsFunc` 联动

### 方案 B：提交后重建表单再选

先接受自定义输入，再在一次提交后重建表单，把新模型加入列表，供用户重新选择。

**优点**：

- 实现简单

**缺点**：

- 交互被打断
- 不符合“失焦后立刻出现”的要求

### 方案 C：只改前缀样式，不改保存流

保留当前 `mergeProviderModels()` 的双来源合并方案，仅把前缀从圆点改成方框。

**优点**：

- 改动最小

**缺点**：

- 无法解决“输入后不能立刻出现在列表中”的主诉求

**结论**：采用方案 A。

## 设计

### 1. 状态模型

`editProvider()` 中的模型状态拆成三层：

#### 1.1 预设模型集合

来源于 `models.PresetModels`，固定不变。

#### 1.2 自定义模型注册池

用于维护当前 provider 可供选择的自定义模型列表，包含：

- 该 provider 原本已经保存的自定义模型
- 本次编辑过程中，用户新输入并注册的自定义模型

一旦模型进入注册池，在本轮表单结束前持续存在；后续删除输入框文字不会把它移出注册池。

#### 1.3 当前选中模型

与 `MultiSelect` 双向绑定，表示本次最终要保存的模型集合。

**最终保存来源只认这一层。**

### 2. 数据流

#### 现状

```text
selectedModels + customModelInput
    -> mergeProviderModels()
    -> finalModels
```

#### 改造后

```text
preset models + custom registry
    -> dynamic options for MultiSelect
    -> selectedModels
    -> finalModels
```

输入框不再直接进入 `finalModels` 计算，只负责把新模型注册进 options，并自动选中。

### 3. 表单字段职责

保留现有 5 个字段顺序：

1. Provider 标识名
2. Base URL
3. API Key
4. 模型列表（`MultiSelect`）
5. 自定义模型名称（`Input`）

其中第 4、5 项职责调整如下：

#### 3.1 模型列表字段

由静态 `.Options(...)` 改为动态 `.OptionsFunc(...)`，内容由“预设模型 + 自定义模型注册池”共同生成。

要求：

- 预设模型始终在前
- 自定义模型追加在后
- 已选中的模型进入 options 时应带 `.Selected(true)`
- 重复项不得重复渲染

#### 3.2 自定义模型输入框

改为“新增入口”，在用户完成一次输入动作后执行注册逻辑。

**用户可见行为约束（本次设计固定）**：

- 当用户在该字段中输入一个或多个模型名，并以“完成输入”的方式离开当前编辑（如 `enter/tab`）时，应尝试注册这些模型
- 若本次输入被系统识别并处理（无论是成功注册出至少一个新模型，还是输入内容全部已存在而未新增），则：
  - 若存在全新模型，新模型立即出现在上方模型列表中，并默认自动选中
  - 输入框的**可见内容**被清空，而不只是绑定值被清空
  - 焦点返回模型列表，让用户继续勾选/取消
  - 本次操作**不直接提交**整个 Provider 表单
- 若本次输入为空，或解析后没有任何有效模型名，则不触发注册流程，可沿用正常导航语义
- 删除输入框中的文字不会移除已经注册到列表中的模型

**实现自由度（本次设计允许）**：

- 允许使用适量的字段包装、自定义 bubbletea 消息、或局部表单导航控制，来实现上述用户可见行为
- 不要求把注册逻辑强行绑定到某个特定底层钩子（如仅通过 `Accessor.Set` 或仅通过 `Blur()`）
- 只要最终体验满足本节约束，底层实现细节可在实现计划阶段确定

注册步骤如下：

1. 解析输入内容（支持逗号、中文逗号、分号、换行）
2. 将全新模型加入自定义模型注册池
3. 将新增模型加入 `selectedModels`
4. 清空输入框的可见文本与绑定值
5. 返回模型列表，供用户继续选择

该字段不承担删除或反向同步列表的职责。

### 4. 运行时交互

#### 场景 A：新增一个自定义模型

```text
输入 custom-alpha
-> 按 enter/tab 离开输入框
-> custom-alpha 出现在模型列表底部
-> custom-alpha 默认选中
-> 输入框清空
```

#### 场景 B：一次输入多个模型

```text
输入 custom-alpha, custom-beta
-> 离开输入框
-> 两项都加入列表
-> 两项都默认选中
-> 输入框被清空
-> 焦点返回模型列表，继续勾选/取消
```

#### 场景 C：最后一个字段完成输入

```text
输入 custom-alpha
-> 在最后一个输入框按 enter/tab
-> custom-alpha 被注册进列表并默认选中
-> 输入框被清空
-> 不直接提交 Provider 表单
-> 焦点返回模型列表，等待用户继续选择
```

#### 场景 D：删除输入框文字

```text
输入框清空
-> 不影响已经加入列表的 custom-alpha
-> custom-alpha 是否保存，仅取决于列表是否仍勾选
```

#### 场景 E：取消某个已加入的自定义模型

```text
custom-alpha 已在列表中
-> 用户在 MultiSelect 中取消勾选
-> 保存时 custom-alpha 不进入 finalModels
-> 但列表项仍保留，可重新勾选
```

### 5. 样式设计

用户已确认采用 ASCII 方框样式：

- 未选中：`[ ] `
- 已选中：`[✓] `

`huh v0.6.0` 的 `MultiSelect` 前缀由 `Theme.Focused.SelectedPrefix` / `UnselectedPrefix` 控制，因此样式修改优先通过主题层完成，而不是修改字段文本本身。

#### 样式范围

本次设计**固定采用字段级局部主题覆盖**，只影响 `editProvider()` 中的 provider 模型列表这一处 `MultiSelect`，避免误伤其他 `MultiSelect` 页面。

不允许在实现阶段再把该决策扩展为全局主题改动；若未来要统一全局样式，应作为单独需求处理。

### 6. 组件与辅助函数调整

#### 保留不变

- `splitProviderModelsForEdit()`：仍可用于把已有 provider 的模型拆成“预设已选”和“已有自定义模型”初始值
- `parseCustomModelInput()`：继续承担多分隔符解析与去重职责
- `appendUniqueStrings()`：继续用于有序去重追加

#### 计划新增或重组的辅助逻辑

建议新增一组围绕动态模型列表的轻量函数，职责如下：

- 从“预设模型 + 自定义注册池 + 当前选中结果”构建 `[]huh.Option[string]`
- 将输入框内容注册进自定义模型池并同步到 `selectedModels`
- 判断某个模型是否为预设模型，以避免与预设重名时重复追加自定义项
- 判断一次输入是否产生“有效新增模型”，以决定是否清空输入框并返回模型列表

这些函数只服务 `editProvider()`，不扩散到 `internal/config/` 或其他步骤页面。

### 7. 边界条件

#### 7.1 重复输入

若输入 `custom-alpha, custom-alpha`，最终仅保留一个 `custom-alpha` 列表项。

#### 7.2 与预设模型重名

若输入内容与预设模型同名（例如 `gpt-5.2`）：

- 不创建额外自定义项
- 仅确保该预设模型被加入 `selectedModels`

#### 7.3 编辑已有 provider

当用户编辑一个已保存过自定义模型的 provider 时：

- 这些自定义模型在表单初始渲染时就应出现在列表中
- 它们是否初始选中，取决于原 `p.Models`
- 未勾选但已注册的自定义模型，只保证在当前这一次 `editProvider()` 会话内保留；若本次未被勾选保存，则下次重新进入编辑时不要求恢复

#### 7.4 至少选择一个模型

最终验证规则保留：必须至少有一个模型进入 `selectedModels`。

但验证来源从“预设勾选 + 输入框文本组合判断”收敛为“最终勾选结果是否为空”。

## 测试策略

### 单元测试

新增或调整测试，验证以下行为：

1. 已有 provider 模型能正确拆分成“预设已选 + 自定义注册池初始值”
2. 动态 options 构建结果顺序正确（预设在前，自定义在后）
3. 注册新自定义模型后，列表包含该项且默认选中
4. 重复注册不会产生重复 option
5. 输入预设同名模型时不会生成重复项
6. 最终保存结果只依赖 `selectedModels`

### 回归验证

手动验证以下路径：

1. 新建 provider，输入一个全新自定义模型，确认立即出现在列表中
2. 新建 provider，一次输入多个全新模型，确认全部立即出现且默认选中
3. 在最后一个输入框按 `enter/tab` 注册成功后，确认不直接提交表单，而是返回模型列表
4. 编辑已有 provider，输入一个已存在的模型名，确认不会重复新增列表项
5. 编辑已有 provider，确认历史自定义模型初始可见
6. 取消某个自定义模型勾选后保存，确认该模型未进入结果
7. 删除输入框内容后，确认已加入列表的自定义模型仍保留
8. 观察多选前缀已从圆点改为 `[ ]` / `[✓]`
