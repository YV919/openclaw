# Provider 模型列表可见高度设计文档

**日期**：2026-03-19
**状态**：待实施
**模块**：`app.go`

---

## 背景与问题

当前 `editProvider()` 中的“模型列表”字段使用 `huh.NewMultiSelect[string]()` 渲染 provider 支持的模型选项：

- 预设模型来自 `internal/models/presets.go`
- 已保存的自定义模型会被合并进动态 options
- 底部另有统一固定帮助栏：`ctrl+c 取消  ·  shift+tab 上一项  ·  enter 确认`

用户在真实终端中反馈的问题是：

1. **列表不一定能完整显示**：当模型数量较多、终端高度较小、且页面上还有其他输入字段与底部固定帮助栏时，模型列表可能只显示其中一部分。
2. **缺少剩余提示**：当前即使还有模型项未显示，界面也不会明确说明“还有多少项未显示”，用户只能靠滚动自行猜测。
3. **主诉优先级已确认**：用户明确希望优先采用“**尽量显示全**”；如果确实无法显示全，也要明确提示还有多少模型未显示。
4. **改动范围已确认**：本次只修改 provider 编辑页中的“模型列表”这一处，不扩大到主/子 Agent 等其他 Select 列表。

### 已核实的 huh v0.6.0 事实

基于当前项目实际依赖 `github.com/charmbracelet/huh v0.6.0` 源码确认：

- `field_multiselect.go:173-176`：`Limit(limit int)` 用于限制**最多可选择多少项**，不是控制可见行数。
- `field_multiselect.go:180-185`：`Height(height int)` 才用于设置字段高度。
- `field_multiselect.go:456-467`：`MultiSelect` 的可见视口高度由 `height - titleHeight - descriptionHeight` 计算得出；未设置高度时，视口高度默认等于 options 总数。
- `form.go:510-527` 与 `group.go:270-272`：表单和分组会在收到 `tea.WindowSizeMsg` 后根据终端窗口高度压缩内容，因此即便字段本身未显式限制高度，实际渲染时仍可能因 group/form 高度收缩而显示不全。

结论：**问题不是模型数据丢失，而是终端空间收缩时，模型列表缺少可预期的高度策略与剩余提示。**

---

## 目标

在不改变 `editProvider()` 页面整体结构和字段顺序的前提下，实现以下体验：

1. provider 的“模型列表”字段应优先**尽量显示更多模型项**。
2. 当当前终端空间足够时，应能**直接显示全部模型**。
3. 当当前终端空间不足时，应明确提示：**当前仅显示前 N 项，还有 M 项可继续向下查看**。
4. 保持现有的模型勾选、自定义模型动态注册、ASCII 复选框样式、统一帮助栏等交互不变。

---

## 方案比较

### 方案 A（推荐）：动态计算 MultiSelect 高度 + 动态剩余提示

仅对 provider 的 `MultiSelect` 做定向增强：

- 运行时根据当前终端高度、页面其他字段占用高度、固定帮助栏高度，计算该模型列表的一个“安全可用高度”。
- 将计算结果用于 `MultiSelect.Height(...)`，让列表尽可能多显示选项。
- 同时用 `DescriptionFunc(...)` 动态生成说明文案：
  - 能全部显示时，只显示基础说明。
  - 不能全部显示时，追加“当前仅显示前 N 项，还有 M 项可继续向下查看”。

**优点**：

- 完全符合用户确认的交互目标
- 改动范围小，只触及 provider 模型列表
- 与 `huh v0.6.0` 真实 API 语义一致
- 在不同终端尺寸下表现自适应

**缺点**：

- 需要引入终端尺寸感知与字段高度计算逻辑
- 需要避免描述文本变长后反过来挤压列表视口，形成循环依赖

### 方案 B：固定高度 + 剩余提示

无论终端大小，始终给模型列表一个固定高度（例如 8 或 10 行），并在有溢出时追加剩余数量提示。

**优点**：

- 实现简单
- 界面稳定，不随终端大小变化太多

**缺点**：

- 不符合“尽量显示全”的优先诉求
- 在大终端中浪费可用空间

### 方案 C：仅补充剩余提示，不干预高度

保留现有渲染逻辑，只在描述中补上剩余数量。

**优点**：

- 改动最小

**缺点**：

- 不能改善“能显示更多却没有显示”的体验
- 只解释症状，不改善展示效果

**结论**：采用方案 A。

## 设计

### 1. 作用范围

仅修改 `editProvider()` 中的这个字段：`app.go:686-700`。

不改动：

- Step 2 主 Agent 模型选择
- Step 3 子 Agent 模型选择
- 命名 Agent 模型选择
- `internal/config/`、`internal/models/` 目录内容

### 2. 用户可见行为

#### 2.1 终端空间足够时

- 模型列表直接显示全部选项。
- Description 保持为现有说明文案，不额外显示“还有 N 项”。

#### 2.2 终端空间不足时

- 模型列表仍保持可滚动。
- 界面显示尽可能多的可见项。
- Description 在原有说明后追加一行，例如：

```text
当前仅显示前 6 项，还有 4 项可继续向下查看
```

其中：

- `6` = 当前可见项数量
- `4` = `总模型数 - 当前可见项数量`

#### 2.3 其他行为保持不变

- 勾选/取消逻辑不变
- 自定义模型注册后仍动态加入列表并默认选中
- ASCII 复选框主题不变
- 统一底部帮助栏不变

### 3. 运行时数据

`editProvider()` 已有：

- `selectedModels`
- `customModelRegistry`
- `modelOptionsVersion`

本次为 provider 模型列表再引入两类运行时信息：

#### 3.1 当前终端窗口高度

用于估算该页可分配给 `MultiSelect` 的最大高度。

来源可为：

- `tea.WindowSizeMsg` 在现有 `runForm` / `formModel` 包装层中缓存的窗口尺寸
- 或其他等价的、与 Bubble Tea 生命周期兼容的窗口高度来源

本设计不强制绑定具体实现方式，但要求高度来源必须来自**当前实际终端窗口**，而非硬编码常量。

#### 3.2 当前模型选项总数

由 `buildProviderModelOptions(selectedModels, customModelRegistry)` 的结果长度决定。

该值用于：

- 计算是否能全部显示
- 计算剩余未显示数量
- 驱动动态 Description

### 4. 高度策略

### 4.1 核心原则

模型列表的高度策略不是“固定几行”，而是：

> **在不破坏当前页面其余字段可用性的前提下，把尽可能多的垂直空间让给模型列表。**

### 4.2 计算结果

需要得到一个整数 `visibleRows`，表示模型列表内部实际可见的选项行数。

约束：

- `visibleRows >= 1`
- `visibleRows <= optionCount`
- 若终端高度足够，则 `visibleRows == optionCount`
- 若终端高度不足，则 `visibleRows < optionCount`

### 4.3 计算所需预留空间

在估算模型列表高度时，必须为以下内容预留空间：

1. Provider 标识名输入框
2. Base URL 输入框
3. API Key 输入框
4. 自定义模型输入框
5. 字段之间的 group 间距
6. 统一底部帮助栏
7. 模型列表自身的标题与描述文字高度

也就是说，模型列表拿到的高度应是**页面剩余空间**，而不是直接拿终端总高度。

### 4.4 避免循环依赖

由于 `MultiSelect.Height(...)` 的视口高度计算会扣除标题和描述高度，而“剩余项提示”本身又会增加描述高度，因此实现时必须避免这种循环：

```text
先加提示 -> 描述变高 -> 可见项变少 -> 提示内容变化 -> 再次影响高度
```

因此本设计要求：

- 高度估算必须基于一套**稳定规则**完成
- 剩余提示最多只追加**一行**
- 计算逻辑应保证在同一窗口高度下，`visibleRows` 收敛为单一稳定值，而不是在相邻值之间抖动

### 5. Description 策略

当前基础说明文案为：

```text
空格/x 切换选中，↑↓ 移动，enter 确认
选择此 provider 支持的预设模型；如需自定义模型，请在下方输入框填写，可一次填写多个
```

改造后使用 `DescriptionFunc(...)` 动态生成：

#### 5.1 全量可见时

仅显示基础说明文案。

#### 5.2 部分可见时

在基础说明后追加一行剩余提示：

```text
当前仅显示前 N 项，还有 M 项可继续向下查看
```

要求：

- 文案为确定性输出，不依赖用户当前位置或滚动偏移
- 只说明“当前最大可见能力”，不展示复杂分页信息
- 不引入额外操作提示（如 PgUp/PgDn），避免新增学习成本

### 6. 代码结构建议

### 6.1 保持 editProvider 的主流程不变

`editProvider()` 仍继续负责：

- 初始化 provider 基础字段
- 构造 `MultiSelect`
- 构造自定义模型输入框
- 提交后保存最终模型列表

### 6.2 新增局部辅助逻辑

建议把“provider 模型列表可见高度与提示计算”封装成局部辅助函数，职责与现有 `buildProviderModelOptions()`、`finalProviderModels()` 类似，避免把高度公式直接内联在链式表单定义中。

辅助逻辑至少要能回答两个问题：

1. 当前窗口下，模型列表应给多少 `Height(...)`
2. 当前窗口下，Description 是否要追加“还有 N 项”提示

### 6.3 不扩散到 runForm 的通用语义

虽然窗口尺寸大概率会在 `runForm` / `formModel` 层获取，但本次需求是 provider 模型列表的定向体验优化，因此不应把“长列表自动高度策略”抽象成作用于所有字段的通用框架。

原因：

- 用户当前只要求修复 provider 模型列表
- 其他表单未确认需要相同行为
- 过早抽象会扩大测试面和回归风险

## 测试与验证

### 手动验证场景

#### 场景 A：大终端窗口

- 打开 provider 编辑页
- 模型总数为当前预设数 + 已注册自定义模型数
- 若窗口足够高，应直接看到全部模型项
- Description 不应出现“还有 N 项”提示

#### 场景 B：小终端窗口

- 打开 provider 编辑页
- 模型列表只显示部分项
- Description 应追加“当前仅显示前 N 项，还有 M 项可继续向下查看”
- 用户仍可通过上下键滚动访问剩余项

#### 场景 C：动态注册自定义模型后

- 在自定义模型输入框新增 1 个或多个模型
- 模型列表总数变大后，提示内容应同步更新
- 新模型仍默认选中
- 原有注册行为不受破坏

#### 场景 D：边界窗口高度

- 当窗口非常小但仍能渲染表单时
- 模型列表至少保留 1 行可见项
- 不应出现负高度、空白块异常或崩溃

### 自动化测试建议

本次主要是 TUI 布局行为调整，不强制要求完整端到端 TUI 测试；但应补充能覆盖纯函数/纯计算逻辑的单元测试，例如：

- 在给定窗口高度与选项数量下，计算出的 `visibleRows` 是否符合预期
- 是否正确判断“全量可见 / 部分可见”
- 剩余提示文案是否正确生成

若实现中提取了纯辅助函数，则这些测试应优先落在辅助函数上，而不是对 Bubble Tea 交互做脆弱快照测试。
