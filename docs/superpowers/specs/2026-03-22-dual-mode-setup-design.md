# 快速配置 / 高级配置双模式设计文档

**日期**：2026-03-22
**状态**：待实施
**模块**：`app.go`

---

## 背景与问题

当前 `openclaw-config` 的启动流程只有一条完整向导路径：

1. Provider 管理
2. 主 Agent 模型
3. 子 Agent 模型
4. 命名 Agent 管理

这条路径适合熟悉 OpenClaw 配置结构的用户，但对小白用户来说过长，且需要理解：

- Provider 与模型的关系
- 主 / 子 / 命名 Agent 的区别
- 哪些配置可以继承、哪些配置需要单独指定

用户明确提出需要在程序一开始提供两个选择：

- **快速配置**：面向小白用户，最终只保留一个 Provider，并选择一个主模型，然后结束配置；其他模型全部跟主模型相同
- **高级配置**：保留现有完整配置能力

在进一步澄清后，用户确认“其他模型跟主模型相同”应采用当前项目已有的**默认继承语义**，而不是把所有模型都显式写成同一个值：

- 主 Agent：显式配置
- 子 Agent：留空表示跟随主 Agent
- 命名 Agent：不单独配置

此外，用户还补充了一个关键交互要求：

- 如果当前已经存在高级配置（例如子 Agent 或命名 Agent 的自定义模型），进入快速配置时不能直接永久覆盖
- 应先做**临时备份**，再把高级配置清空为默认继承状态
- 在快速配置结束时，要明确告知用户当前情况，并提供：
  - **完成配置**
  - **恢复原样**

也就是说，本次需求并不是单纯地给现有流程加一个入口，而是要增加一条**安全、可回退的短流程配置路径**。

## 已核实的实现事实

基于当前代码，以下事实已经确认：

1. 程序入口在 `app.go:34` 的 `(*App).Run()` 中。
2. 启动时会先：
   - 加载配置 `LoadFullConfig()`
   - 检查更新
   - 打印 banner
3. 当前主流程通过 `step := 1` 到 `step <= 4` 的循环驱动：
   - `runStep1Providers()`
   - `runStep2MainAgent()`
   - `runStep3SubAgent()`
   - `runStep4NamedAgents()`
4. 最终写盘只发生在 `Run()` 末尾的 `SaveFullConfig(fullCfg)`。
5. 当前配置结构 `internal/config/types.go` 中，`SubAgent` 的注释已明确：
   - `Primary 空 = 同主 agent`
6. 命名 Agent 也已经支持“同主 Agent”语义：在命名 Agent 编辑页中，`Primary` 可为空。
7. 因为 `Run()` 在保存前始终操作内存中的 `fullCfg`，所以可以在本次运行内安全地：
   - 创建临时快照
   - 清空高级配置
   - 在需要时恢复进入快速模式前的整份配置
8. 高级流程目前已经稳定工作，本次需求应尽量避免破坏现有 Step 2~4 的行为。

## 目标

在不修改底层配置结构、不改变现有高级配置能力的前提下，实现以下目标：

1. 程序启动后先让用户选择：
   - 快速配置
   - 高级配置
2. 快速配置只让用户完成：
   - 单 Provider 配置
   - 主 Agent 模型选择
3. 进入快速配置时，将进入前的整份配置只在**内存中**做临时备份。
4. 进入快速配置后，将高级配置切换为默认继承态：
   - `SubAgent` 清空
   - `NamedAgents` 清空
   - 最终 Provider 列表收敛为用户在快速配置中确认的 1 个 Provider
5. 快速配置完成后，明确提示用户：
   - 原配置已临时备份
   - 当前子 Agent / 命名 Agent 已改为默认继承主 Agent
   - 可选择完成配置或恢复原样
6. 用户只有在选择“完成配置”后，当前快速配置结果才会进入最终保存。
7. 高级配置入口保留现有 4 步完整流程，不引入行为变化。

---

## 非目标

本次不做以下事情：

1. 不新增新的配置文件或磁盘级备份文件。
2. 不改造 `internal/config/manager.go` 或 `internal/config/types.go` 的底层结构。
3. 不改变现有高级配置每一步的字段顺序、字段含义或保存格式。
4. 不把快速配置扩展成单页表单；仍保持步骤式交互。
5. 不让快速配置直接复用现有高级模式的整页步骤语义；快速模式可以复用底层表单能力与数据处理函数，但必须拥有专用步骤。

## 方案比较

### 方案 A（推荐）：入口分流 + 独立快速流程 + 运行时快照恢复

在 `Run()` 中、banner 之后、现有 step loop 之前新增一个“配置模式选择”页：

- 快速配置
- 高级配置

其中：

- 快速配置走独立短流程：
  1. 创建快照
  2. 清空高级配置
  3. 使用快速配置专用的单 Provider 配置
  4. 使用快速配置专用的主模型选择页
  5. 进入结果确认 / 恢复页
- 高级配置继续走现有 4 步循环

**优点**：

- 最符合用户心智，“快速”和“高级”从入口就明确分开
- 高级流程几乎不需要改动，回归风险低
- 快速流程可以单独维护自己的单 Provider 语义与回退逻辑
- 后续如果要继续简化快速流程，扩展性更好

**缺点**：

- 需要新增一个短流程函数和一份临时快照结构

### 方案 B：在现有 step loop 里加一个 quick mode 开关

先新增一个选择页，选快速后只跑 Step 1 + Step 2，再插入确认页；选高级则继续跑 1~4 步。

**优点**：

- 改动表面上集中在 `Run()` 循环中

**缺点**：

- 会让 step loop 同时承担两条流程的导航逻辑
- 返回上一步、结束、恢复原样等行为会变得更绕
- 可读性和后续维护性较差

### 方案 C：快速配置直接覆盖所有高级配置，不提供恢复

**优点**：

- 实现最简单

**缺点**：

- 明确违背用户要求的“临时备份 + 可恢复”
- 小白用户试配置时风险过高

**结论**：采用方案 A。

## 设计

### 1. 启动入口

在 banner 之后新增一个模式选择页，文案明确说明两条路径的差别：

- **快速配置**：只配置 Provider 和主模型，其他 Agent 默认继承主模型
- **高级配置**：完整配置主 Agent、子 Agent、命名 Agent

这个选择发生在任何 Step 1~4 之前。

### 2. 运行时快照

为快速配置引入一个运行时临时快照，用于保存进入快速模式前的完整配置状态。

建议快照至少包含：

- `Providers`
- `MainAgent`
- `SubAgent`
- `NamedAgents`

即使用户重点提到的是“高级配置”，也应把 `Providers` 与 `MainAgent` 一并纳入快照。原因是：

- 用户在快速配置中会收敛 Provider 到单个结果
- 用户在快速配置中同样会修改主模型
- 如果最终选择“恢复原样”，用户预期应回到进入快速模式前的完整配置状态

快照只存在于本次运行内，不落盘。

### 3. 快速配置的初始化行为

进入快速配置后，立即进行以下动作：

1. 创建整份配置快照
2. 将当前 `fullCfg` 切换到快速配置的初始态：
   - `fullCfg.SubAgent = config.AgentModelConfig{}`
   - `fullCfg.NamedAgents = nil`

这里不额外弹确认框，原因是：

- 当前更改只发生在内存里
- 后面仍然有“恢复原样”出口
- 小白用户的快速路径应尽量短，不应一开始就被复杂确认打断

### 4. 快速配置流程

快速配置建议拆成独立函数，例如概念上：

- `runQuickSetup(...)`

其内部按三步执行：

#### 4.1 Quick Step 1：单 Provider 配置

快速配置应使用一个专用步骤，例如概念上的 `runQuickStep1Provider(...)`。该步骤目标不是提供完整的 Provider 管理能力，而是让用户尽快完成一个可用 Provider 的完整配置。

行为要求：

- 只保留一个最终 Provider
- 如果当前已有多个 Provider，进入该步骤后用户最终确认的结果应收敛为 1 个
- 不要求用户理解“多个 Provider 并存”的高级配置概念
- 完成该步骤后，`fullCfg.Providers` 仅保留用户最终确认的那 1 个 Provider
- 该步骤可以复用现有 Provider 编辑所需的底层字段、校验和表单组件，但不能直接复用 `runStep1Providers()` 的“Provider 管理”整页语义

#### 4.2 Quick Step 2：主模型选择

快速配置使用一个专用步骤，例如概念上的 `runQuickStep2PrimaryModel(...)`，只暴露一个“主模型”选择框。

该步骤与高级配置的区别必须明确：

- 只允许用户选择一个主模型
- 只允许从 Quick Step 1 中最终确认的那个 Provider 的 `models` 列表中选择
- 不展示 `Fallback`
- 不允许在这一步临时新增 model
- 如果该 Provider 没有任何 models，则这一步不能完成，并提示用户返回上一步补充 Provider models
- 保存时写入：
  - `fullCfg.MainAgent.Primary = selectedProvider.Name + "/" + selectedModel`
  - `fullCfg.MainAgent.Fallback = ""`

这样可以保证小白用户完成快速配置时，看到的配置项数量与实际保存语义完全一致，不会在“只想选一个主模型”时被额外的备用模型概念打扰。

#### 4.3 Quick Step 3：结果确认 / 恢复页

新增一个专用确认页，负责向用户说明：

- 原配置已经临时备份
- 当前子 Agent 与命名 Agent 已重置为默认继承主 Agent
- 当前主 Agent 只保留一个主模型，不再额外配置备用模型
- 当前快速配置最终只会保留一个 Provider
- 可以选择：
  - **完成配置**
  - **恢复原样**
  - **返回上一步**

行为约束：

- 选 **完成配置**：保留当前快速配置结果，结束快速流程
- 选 **恢复原样**：把快照中的完整配置恢复回 `fullCfg`，结束快速流程
- 选 **返回上一步**：回到快速配置专用的主模型选择页

### 5. 代码结构拆分

实现时应明确区分“流程控制”和“底层复用”：

- `Run()` 只负责加载配置、显示 banner、选择模式、分流到快速/高级流程、最后统一保存
- `runAdvancedSetup(...)` 保持现有四步循环，不混入快速模式判断
- `runQuickSetup(...)` 拥有自己的 2+1 步流程：
  1. 单 Provider 配置
  2. 主模型选择
  3. 完成/恢复页

推荐的函数边界：

- `pickSetupMode(...)`
- `prepareQuickSetup(...)`
- `restoreQuickSetupSnapshot(...)`
- `runQuickStep1Provider(...)`
- `runQuickStep2PrimaryModel(...)`
- `pickQuickSetupAction(...)`

其中：

- 快速模式可以复用构建 model options、Provider 字段校验、帮助文案渲染、快照恢复等底层能力
- 快速模式不应直接复用 `runStep1Providers()` 与 `runStep2MainAgent()` 这类带有高级模式整页语义的步骤函数
- Quick Step 2 只围绕 Quick Step 1 最终确认的那个 Provider 构建模型选项，不能从全局 Provider 池生成 `provider/model` 列表

### 6. 高级配置流程

高级配置继续使用当前 step loop：

1. Provider 管理
2. 主 Agent 模型
3. 子 Agent 模型
4. 命名 Agent 管理

本次需求不改变高级流程中的字段、文案、继承语义和保存行为。`Fallback` 仍然只属于高级配置能力的一部分。

### 7. 数据规则与最终保存边界

快速配置完成后，内存中的 `fullCfg` 应落在以下稳定形态：

- `Providers`：只保留 1 个 Provider
- `MainAgent.Primary`：显式设置为所选主模型（格式为 `provider/model`）
- `MainAgent.Fallback`：清空
- `SubAgent.Primary`：清空
- `SubAgent.Fallback`：清空
- `NamedAgents`：清空

也就是说：

- 主 Agent 显式使用用户选择的模型
- 子 Agent 通过空值语义默认继承主 Agent
- 命名 Agent 不再保留单独配置

无论用户走快速配置还是高级配置，都继续复用 `Run()` 末尾统一的 `SaveFullConfig(fullCfg)`。

关键约束：

- 快速配置期间的所有修改都只存在于内存中的 `fullCfg`
- 只有当快速流程选择“完成配置”后，这份 `fullCfg` 才会进入最终保存
- 如果进入快速模式前 `MainAgent.Fallback` 有值，则用户选择“完成配置”时必须显式清空为 `""`
- 如果用户在快速流程中选择“恢复原样”，则应先恢复整份快照，再走统一保存逻辑

这保证：

- 快速配置是“可试配”的
- 用户不会因为进入快速模式而意外永久丢失原高级配置

### 8. 成功页输出

当前 `printSuccess(cfg)` 会根据 `cfg` 内容输出 Provider、主 Agent、子 Agent、命名 Agent 摘要。

在快速配置完成并保存后，成功页应自然反映出：

- 主 Agent 已配置
- 子 Agent 未单独配置（即不显示独立值，等价于默认继承）
- 命名 Agent 为空

可选地，可在成功提示文案层面补充一句说明，强调：

- 当前子 Agent / 命名 Agent 均使用默认继承主 Agent

如果实现复杂度高，也可以先保持 `printSuccess()` 的结构不变，只依赖现有输出结果体现“未单独配置”的状态。

## 数据流

### 快速配置完成路径

```text
加载 fullCfg
-> 用户选择“快速配置”
-> 创建运行时快照(snapshot)
-> 清空 SubAgent / NamedAgents
-> 单 Provider 配置
-> 主模型选择
-> 结果确认页选择“完成配置”
-> SaveFullConfig(fullCfg)
-> 成功页
```

### 快速配置恢复路径

```text
加载 fullCfg
-> 用户选择“快速配置”
-> 创建运行时快照(snapshot)
-> 清空 SubAgent / NamedAgents
-> 单 Provider 配置
-> 主模型选择
-> 结果确认页选择“恢复原样”
-> 用 snapshot 恢复完整配置
-> SaveFullConfig(fullCfg)
-> 成功页
```

### 高级配置路径

```text
加载 fullCfg
-> 用户选择“高级配置”
-> Step1 Providers
-> Step2 MainAgent
-> Step3 SubAgent
-> Step4 NamedAgents
-> SaveFullConfig(fullCfg)
-> 成功页
```

## 边界行为

### 1. 快速配置中途取消

如果用户在快速配置任意一步按 `ctrl+c` / 取消：

- 保持现有全局语义：直接退出程序，不保存
- 因为变更尚未落盘，所以磁盘配置不会受影响

### 2. 快速配置里恢复原样后继续保存

恢复动作应恢复的是“进入快速模式前的完整配置状态”，而不是只恢复部分字段。

因此恢复时应一次性恢复：

- `Providers`
- `MainAgent`
- `SubAgent`
- `NamedAgents`

### 3. Provider 在快速配置中被改动后再恢复原样

如果用户在快速配置中删除、替换或新建了 Provider，再选择“恢复原样”，则应直接恢复进入快速配置前的完整 Provider 列表与 Agent 配置。

原因：

- “恢复原样”的用户预期就是回到进入快速模式前的完整状态
- 既然已经有运行时快照，就不需要额外猜测哪些字段该回滚、哪些不该回滚
- 这也避免了 Provider 已变动但旧 Agent 引用无法对齐的中间态

### 4. 快速配置没有现有高级配置

如果进入快速配置前本来就没有子 Agent / 命名 Agent 自定义项，流程仍然一致：

- 仍创建快照
- 仍清空到默认继承态（结果等价于不变）
- 最终确认页仍可展示“当前高级配置已为默认继承”语义

## 测试策略

本次按 TDD 推进，优先测试“快速模式的数据收敛与状态转换”，再补页面行为测试。

建议的测试顺序：

1. **快照与恢复测试**
   - `prepareQuickSetup(...)`：给定包含多个 `Providers` 和复杂 `MainAgent/SubAgent/NamedAgents` 的 `FullConfig`，断言：
     - 快照保存了原始 `Providers/MainAgent/SubAgent/NamedAgents`
     - 当前配置被切到快速模式初始态
     - `SubAgent` 被清空
     - `NamedAgents` 被清空
   - `restoreQuickSetupSnapshot(...)`：断言快速模式改写过的配置可以完整恢复到进入快速模式前的状态

2. **快速模式最终收敛测试**
   - 完成快速配置后：
     - 只保留 1 个 Provider
     - `MainAgent.Primary` 为 `provider/model` 格式的所选模型
     - `MainAgent.Fallback` 为空
     - `SubAgent` 为空（默认继承）
     - `NamedAgents` 为空

3. **主模型来源限制测试**
   - 快速模式主模型选项只能来自 Quick Step 1 最终确认的那个 Provider 的 `models`
   - 不包含其他 Provider 的模型
   - Provider 没有 models 时，这一步不能完成，并提示返回上一步补充 models

4. **恢复原样测试**
   - 在快速配置中修改 Provider 与主模型后选择恢复原样
   - 断言 `Providers/MainAgent/SubAgent/NamedAgents` 全部恢复到原值

5. **页面级回归测试**
   - 模式入口存在“快速配置 / 高级配置”及清晰文案
   - 快速完成页存在：
     - 完成配置
     - 恢复原样
     - 返回上一步
   - 快速完成页文案明确说明“其他 Agent 默认继承主模型”

6. **高级流程回归测试**
   - 确认现有 Step 2 / Step 3 / Step 4 的行为未被破坏
   - 特别是：
     - 子 Agent “同主 Agent”逻辑
     - 命名 Agent 编辑 / 删除逻辑

## 实施建议

建议按以下顺序实施：

1. 先写快照/恢复失败测试
2. 再写快速模式最终收敛失败测试
3. 再写“主模型只能来自当前 Provider.models”失败测试
4. 让这些测试先 fail
5. 实现快速模式专用步骤与分流逻辑
6. 通过后补页面文案与高级流程回归测试

## 结论

本次需求应通过“**启动入口分流 + 独立快速流程 + 运行时快照恢复**”来实现。

最终用户体验将变为：

- 小白用户可直接进入快速配置，完成一个 Provider 的完整配置，再从该 Provider 的 models 中选择唯一主模型
- 高级用户继续使用完整 4 步流程
- 已有高级配置在快速模式下不会被无提示永久覆盖，而是可以在结束时恢复原样

这在保持现有配置结构与高级能力稳定的同时，为新手提供了一条更短、更安全、语义更清晰的配置路径。
